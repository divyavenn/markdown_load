These are my notes on a really stellar book called Operating Systems: Three Easy Pieces, which aims to be a sort of Feynman-esque series of lectures on operating systems. This essay doesn’t go into specific Unix commands because there are many guides to those already. Instead, I hope this gives you an intuitive grounding for the most important (and transferable) concepts in this dense and often convoluted topic.

The fundamental job of an operating system is to create, destroy, wait on, suspend, and check the status of its many processes.

A single CPU core can actually only run one process at at time, but if you open up your Task Manager or Activity Monitor, you would see many processes all running (apparently) at the same time.

The OS creates this illusion virtualizing the CPU. All the active processes take turns with the the available time and memory.

At any given time, they are in one of three states: running, ready, or blocked. Blocked means the process is waiting on some external resource (I/O, a lock, a signal). Ready means the process can run as soon as the CPU is available.

An example of OS schedule: process Andy asks the CPU to add 2+3 and becomes blocked waiting for it be printed to the terminal (I/O request). The OS recognizes Andy is not using the CPU and starts running process Betty. While Betty is running, I/O prints 5 to the terminal, making Andy ready again.

Running is self-explanatory, but what does it actually involve? Until a program is run, it’s just inert code on the disk.

First, the OS has to load the code + any static data from the disk into memory (the process’s address space). In early operating systems, this was done eagerly (all at once before running the program). Modern OSes perform the process lazily (loading pieces of code or data only as they are needed) using techniques like paging and swapping.

The OS then allocates stack and heap memory for the program, hands over file descriptors for I/O and error handling, initializes global variables and static variables. All of this establishes the process’s machine state - what part of the machine it can read and update when running. Finally, the OS jumps to the main()routine, handing over control of the CPU!

There are lots of types of schedulers, but they all need a couple things: (1) a list of all the processes (1) a place to hold information about inactive processes. For example, when a process is stopped, its registers will be saved to the OS’s register context. By restoring these registers, the OS can resume running the process.

scheduling algorithms

A fundamental tradeoff when scheduling jobs (individual processes) is response time versus turnaround time. Turnaround time is how long it takes for the job to complete after it arrives. Response time is how long it takes for the OS to schedule the job after it arrives.

If you wanted to optimize for turnaround time, you’d always run the smallest job first, but this means a time-intensive job could get held up indefinitely. If you wanted to optimize for response time, you’d use a Round Robin schedule, where each job runs for a certain time slice before switching over to another one. However, context-switching between jobs has its own cost, so you can’t switch *too* often.

Turnaround time is important for overall performance, but response time is important for interactivity. You want the OS to handle finish all its jobs as quickly as possible, but you don’t want to wait forever for file dialogs to open or for your terminal to list all the files in current directory.

A solution to this (that won the Turing Award) is the multi-level feedback queue. an MLFQ assigns each job a priority level. Each priority level has its own queue, and every job gets placed in a queue. You then use round-robin scheduling within individual queues.

The MLFQ will adjust the priority of job based on its behavior. For example, if a job is a good citizen (uses the CPU in short, polite bursts and surrenders it quickly) the MLFQ will presume it is an interactive process and keep its priority high.

If, instead, a job is a greedy compute-guzzler (uses the CPU intensively for long periods of time) MLFQ will reduce its priority, deeming it a job that can comfortably run in the background without disrupting the user’s experience too much.

Of course, the algorithm doesn’t have any information about a job in the beginning, so it uses an innocent-before-proven guilty heuristic. All jobs are assumed at first to be short, and given highest priority.

If it actually is a short job, it will surrender the CPU quickly (either because it’s finished or blocked, waiting on an external resource.) If not, it slowly moves down the priority queues. The MLFQ decides an allotment 1- the amount of time a job can spend at a given priority level before the scheduler reduces its priority.

There are two problems with this: firstly, jobs at the bottom of the priority queue could starve (the CPU never gets around to them). Secondly, programs are not unilaterally interactive or intensive. Lightweight user interactions can follow heavy simulations.

To fix this, the algorithm periodically boosts the priority of all the jobs in the system. This means that even the lowest priority jobs are guaranteed their share of the CPU in a round-robin fashion; also, it gives any job the chance to redeem itself as lightweight in the eyes of the scheduler.

There’s a lot you can do with this general design. Each priority can have its own time slice and allotment. You can also implement the priority boost in many different ways, such as decay-usage algorithms.

The problem with MLFQ is that it’s not the most resource efficient scheduler. Scheduling is not a trivial process; in a study of Google datacenters, Kanev et al. show that even after aggressive optimization, scheduling uses about 5% of overall datacenter CPU time. Reducing that overhead as much as possible is an important goal in modern scheduler architecture.

One more lightweight approach is a lottery scheduler. Each process has a certain number of “tickets” in the lottery, and one is picked randomly to determine which process runs in each time slice.

A random approach has some notable advantages. First, random often avoids strange corner-case behaviors that a more traditional algorithm may have trouble handling. Second, random heuristics are lightweight. There’s very little per-process accounting and a decision is as easy as spinning an RNG.

The approach actually used by Linux is the Completely Fair Scheduler. Its goal is to divide a CPU evenly among all competing processes. As each process runs, it accumulates virtual runtime (vruntime). CFS will schedule the process with the lowest vruntime to run next.

Picking the next job is simple, but when to switch isn’t. CFS dynamically manages the time slice with various parameters.

CFS also lets you give some processes priority using niceness weightings (that is the official term). Time slice and vruntime get adjusted according to weight such that nicer processes accumulate vruntime faster. Alas in Linux, too, the nice guys finish last.

CFS also keeps track of all its active (running or ready) processes using a red-black tree instead of a list. Processes are ordered in the tree by vruntime, and inserting, deleting, and searching are logarithmic in time.

As you can see, CFS has a lot of the same elements as MLFQ - a weighted round-robin with dynamic time slices - but it’s a more complicated algorithm with more performance optimizations. Some notable advantages over MLFQ are:

  • CFS extends seamlessly to multi-core CPUs

  • it’s self-adjusting (you don’t have to tweak the queues and time slices manually)

  • fairness is mathematically and continually enforced by keeping track of actual per-process CPU usage

creating and replacing processes

As you can see, the CPU schedule changes dynamically. This means that when writing programs with multiple threads, the order in which they execute is non-deterministic by default.

In this simple C++ function, the hello statement will always be printed first, but once the main process is forked all bets are off! The child process or the parent process may end up executing first occurring to the whims of the OS scheduler. We can control this by having the parent wait() on the child, or by artificially putting the child into the blocked state by making it wait on the parent’s output via IPC.

There are two interesting functions that are worth touching on: fork() and exec().

fork creates a parallel version of the same process you’re already running, with the same code, same memory, same file descriptors.

exec totally replaces the current executable, overwriting the loaded code and user space memory (instructions, variables, heap, and stack). However, the file descriptor table, the process ID, and environment all stay the same.

These two little functions together form an extremely powerful basis for programs in a way that is totally unintuitive.

Imagine running the simplest possible program in your shell - list all the files in the current directory and write them to a text file.

What happens under the hood is:

  1. The shell forks

  2. the child shell closes the standard output (fd = 1) and opens out.txt. This updates the file descriptors, so fd = 1 points to the file.

  3. The child then calls exec(”ls”)

  4. The child’s memory is replaced with the ls program code, but the file descriptors and working directory stay the same, so when ls writes to stdout, it actually goes into out.txt.

  5. Meanwhile, the parent shell just calls wait() until the child finishes.

UNIX pipes are implemented similarly. The output of one process is redirected to an-kernel buffer (the pipe) which another process can read from. Chains of commands can be strung together with minimal effort.

There are lots of ways that different processes can communicate with each other, like signals. You’ve probably used Ctrl+C to kill programs before; that’s just a keyboard shortcut to send an interrupt (SIGINT) to the currently running process. Unix provides many ways for processes send signals and process signals.

multithreading

When you write some code, it runs on a single process by default. Even with a super expensive top of the line 12 core CPU, it’ll take the same amount of time to run (assuming there are no other competing processes).

Using threads, you can rewrite your application to run across multiple CPUs in parallel. This way is the work is accomplished much faster.

Also, it prevents the overall program from being blocked just because one thread is. the next chapter of a game can load in the background, for example, while one thread waits on the whether the user wants to kill or befriend the orc.

Of course, you’d get the same advantages by using multiple processes instead of threads, but each process is like its own program; it has its own code and memory space and machine state. Threads share an address space and thus make it easy to share data. In practice, making a thread is like making a function call, except you don’t know when it will run.

All of this sounds great, but it introduces a whole host of its own problems - mainly, synchronizing data across multiple processors.

All CPUs utilize hardware caches. Caches are small, fast storage that provide quick access to data you might need again soon. Main memory holds all of the data but is slower.

Data is cached based on the idea of locality, both temporal and spatial locality. The idea behind temporal locality is that when a piece of data is accessed, it is likely to be accessed again in the near future; imagine variables or even instructions themselves being accessed over and over again in a loop.

The idea behind spatial locality is that if a program accesses a data item at address x, it is likely to access data items near x as well; here, think of a program streaming through an array, or instructions being executed one after the other

Imagine a process is running on CPU Barty. It reads + updates some data, both of which are stored in Barty’s cache. If the process switches over to CPU Morris, we have to retrieve the data from main memory all over again. Even worse, if it’s a write-back cache (meaning the updates are stored in the cache and written to main memory periodically) the update is lost too!

This problem is called cache coherence, but there are many strategies for the people who design chips to deal with it. This introduces a new consideration to scheduling: cache affinity. A multiprocessor scheduler should prefer to keep a process on the same CPU if at all possible so you don’t lose out on performance by having to rebuild a new cache.

People who write programs still have to make sure the various threads of the program don’t update data in an unpredictable way or at the same time. To fix this, we use mutual exclusion primitives (such as locks and mutexes) to synchronize access to shared data structures.
